# 一期-支付核心系统后端详细设计

## 1. 文档定位

本文档用于指导 `payment-core` 后端冻结版 V1 的实现、联调和测试验收，适用对象：

- 后端开发工程师
- 资深架构师
- 测试工程师
- 产品经理

本文档以当前仓库实现为准，强调正式工程结构、数据库对象、主链路编排和已知边界。

## 2. 系统目标

`payment-core` 一期后端目标是完成家政服务支付核心域的支付交易闭环 V1，确保以下动作可以落库、可追踪、可联调：

1. 订单查询
2. 账单查询
3. 支付流水查询
4. 收银台会话查询
5. 支付单查询
6. 订单发起支付
7. 生成账单
8. 生成预付单
9. 收银台参数查询
10. 支付提交
11. 回调收口
12. 主动查单
13. 关闭支付单
14. 轨迹日志查询

## 3. 工程结构规范

后端统一结构：

```text
backend/src/main/java/com/abc123/hsp
├── common
├── controller
├── dto
├── entity
├── mapper
├── service
│   └── impl
└── HomeServicePaymentApplication.java
```

设计约束：

1. `controller` 只负责协议入参与统一响应输出。
2. `service` 负责能力边界定义。
3. `serviceimpl` 负责全部业务编排、校验、状态流转、日志写入。
4. `mapper` 负责数据库访问声明。
5. `mapper.xml` 负责 SQL 维护。
6. `entity` 负责数据库表对象语义沉淀，不承载业务逻辑。

## 5. 核心对象说明

### 5.1 订单 `OrderEntity`

支付域读取订单信息，用于识别：

1. 订单金额
2. 已付金额
3. 服务品类
4. 当前订单状态

### 5.2 账单 `BillEntity`

账单是订单金额与支付动作之间的桥梁，用于确定本次应付金额、剩余待付金额和到期时间。

### 5.3 预付单 `PrepayOrderEntity`

预付单是收银台的直接载体，承载：

1. 用户本次支付金额
2. 收银台标题
3. 收银台状态
4. 关联支付单号
5. 失效时间

### 5.4 支付单 `PaymentOrderEntity`

支付单是后台运营与支付状态追踪的核心对象，承载：

1. 订单号
2. 用户
3. 支付金额
4. 支付方式
5. 支付渠道
6. 渠道流水号
7. 支付状态

### 5.5 支付轨迹对象

| 对象 | 作用 |
| --- | --- |
| `t_payment_attempt` | 一次支付尝试的请求和响应 |
| `t_payment_route_record` | 一次支付路由决策 |
| `t_payment_notify_log` | 一次回调接收与收口结果 |
| `t_payment_event` | 关键业务事件输出 |

## 7. 关键业务流程

### 7.1 创建预付单流程

1. 根据 `orderNo` 读取订单金额、已付金额、客户名称。
2. 若订单尚未有账单，则创建 `t_bill`。
3. 若订单尚未有支付单，则创建 `t_payment_order`。
4. 创建 `t_prepay_order`。
5. 返回预付单号、支付单号、收银台状态。

### 7.2 支付提交流程

1. 根据 `prepayOrderNo` 读取关联支付单。
2. 更新预付单状态为“支付中”。
3. 回写支付单支付方式与渠道。
4. 写入 `t_payment_route_record`。
5. 写入 `t_payment_attempt`。
6. 写入 `t_payment_event`，事件类型为 `PAYMENT_SUBMIT`。
7. 写入一条待回调日志，供详情页追踪。

### 7.3 回调收口流程

1. 根据 `paymentOrderId` 查询支付单详情。
2. 写入 `t_payment_notify_log`。
3. 根据 `tradeStatus` 更新 `t_payment_order.status`。
4. 写入 `PAYMENT_SUCCESS` 或 `PAYMENT_PENDING` 事件。
5. 返回最新支付详情。

### 7.4 关闭支付流程

1. 查询支付单详情。
2. 更新支付状态为 `CLOSED`。
3. 写入 `PAYMENT_CLOSED` 事件。
4. 返回最新支付详情。

## 8. 事务与一致性

### 8.1 当前已加事务的方法

| 方法 | 事务原因 |
| --- | --- |
| `prepay` | 账单、支付单、预付单需要一并落库 |
| `submit` | 预付单状态、支付单、尝试日志、路由日志、事件日志需要同时提交 |
| `callback` | 回调日志、支付状态、事件日志需要一起生效 |
| `close` | 关闭状态与关闭事件需保持一致 |

### 8.2 一致性要求

1. 预付单和支付单必须保持一对一关联清晰。
2. 支付单详情页必须可追溯路由、回调、事件三类轨迹。
3. 关闭支付后不能再被前端误判为成功。
4. 回调成功后应能通过详情页看到状态和轨迹同步变化。

## 9. 当前版本已知边界

当前版本是“可开发、可联调、可冻结的 V1”，但仍有以下增强项未纳入本次必交付范围：

1. 仍需继续补齐更丰富的查询筛选与分页能力；当前已完成支付记录、支付请求、支付处理日志、账单、支付流水、收银台会话的基础筛选下推。
2. 回调验签、防重、冲突收敛仍为后续增强项。
3. 主动查单当前是查询本地支付详情，未接入真实渠道。
4. 超时关闭巡检任务尚未内建。
5. 退款与结算仅保留查询入口，不承载全流程业务。
