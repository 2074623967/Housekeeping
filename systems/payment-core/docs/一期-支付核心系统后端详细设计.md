# 一期-支付核心系统后端详细设计

## 1. 文档定位

本文档是 `payment-core` 在 `Sunday, July 19, 2026` 的后端详细设计文档。

适用对象：

- 后端开发
- 架构师
- 测试工程师
- 产品经理

## 2. 一期后端目标修正

从 `Sunday, July 19, 2026` 开始，一期后端目标拆成两个层次。

### 2.1 第一阶段A

目标：

1. 建立 Spring Boot 独立服务
2. 提供工作台、订单、支付单、退款单、服务者结算单的查询接口
3. 固化支付平台的主对象和接口路径

### 2.2 第一阶段B

目标：

1. 建立支付交易主链路
2. 支持预付单生成
3. 支持收银台参数组装
4. 支持支付提交
5. 支持回调接收
6. 支持主动查单与关闭支付单
7. 支持支付状态收口

## 3. 模块边界修正

`Sunday, July 19, 2026` 当前代码仍只覆盖查询型接口，这只能视为第一阶段A。

如果要达到完整支付一期标准，第一阶段B必须纳入真实支付主链路建模。

当前已覆盖：

- `DashboardController`
- `OrderController`
- `PaymentController`
- `RefundController`
- `SettlementController`

第一阶段B 需新增：

- 支付下单接口
- 预付单接口
- 支付回调接口
- 查单接口
- 关闭支付接口

## 8. 业务流说明修正

### 8.1 当前已完成：查询流

1. 前端进入页面
2. 调用查询接口
3. Controller 接收请求
4. ServiceImpl 编排业务逻辑
5. Mapper / mapper.xml 查询 MySQL
6. 返回统一响应结构

### 8.2 第一阶段B 需补的支付主链路

1. 用户/客户端发起支付
2. 交易系统传入订单信息
3. 支付核心生成账单与预付单
4. 支付核心返回收银台参数
5. 客户端选择支付方式并提交支付
6. 支付核心发起渠道请求
7. 渠道同步返回受理结果
8. 渠道异步回调支付结果
9. 支付核心幂等收口状态
10. 更新订单、账单、支付单
11. 触发后续账务/清分事件

## 10. 可扩展性要求

### 10.1 幂等

真实支付、退款、结算接口必须具备幂等能力。

### 10.2 审计

所有关键资金操作必须记录：

- 操作人
- 操作时间
- 操作前后值
- 来源系统

### 10.3 重试与补偿

必须支持：

- 查询失败补偿
- 回调丢失补偿
- 状态不一致补偿

### 10.4 回调收口

第一阶段B 必须补：

- 回调日志落库
- 回调验签
- 回调防重
- 状态收口
- 主动查单兜底

## 11. 当前阶段结论修正

从资深架构视角看，`Sunday, July 19, 2026` 当前后端完成的是“支付后台骨架”，还没有完成“真实支付闭环”。

下一步最重要的不是继续加查询页，而是补：

- 账单
- 预付单
- 收银台参数
- 支付提交
- 回调收口
- 查单与关闭

在这之后，`payment-core` 才能真正称为完整的一期支付核心系统。

## 12. 同类产品对标要求

从产品经理和财务架构视角，`payment-core` 后续实现必须对标市面成熟支付产品的共性能力，至少补齐以下设计基线：

1. 支付单、预付单、账单三层对象拆分清晰，避免前台支付对象与后台账务对象混用。
2. 支付状态、回调状态、账单状态分别建模，避免一个状态字段承担全部语义。
3. 后台详情页支持查看支付轨迹、渠道轨迹、回调轨迹、操作轨迹。
4. 支持主动查单、超时关闭、回调防重、查单与回调冲突收敛。
5. 支持后续扩展渠道路由、失败切换、告警巡检、审计留痕。

## 13. 测试经理视角交付要求

后端交付不能只到“接口可调通”，还必须同步交付测试基线：

1. 主流程测试：订单发起支付 -> 预付单 -> 收银台 -> 提交支付 -> 回调成功。
2. 异常流测试：订单不存在、预付单不存在、支付单已关闭、回调重复。
3. 补偿流测试：回调缺失后主动查单、超时关闭、查单成功后二次收口。
4. 数据一致性测试：支付单、账单、订单、事件日志状态一致。
5. 回归测试：支付列表、支付详情、退款列表、结算列表查询能力不回退。
