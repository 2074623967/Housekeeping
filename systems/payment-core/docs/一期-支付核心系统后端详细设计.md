# 一期-支付核心系统后端详细设计

## 1. 文档定位

本文档是 `一期：支付核心系统 MVP` 的后端详细设计文档。

适用对象：

- 后端开发
- 架构师
- 测试工程师
- 产品经理

## 2. 一期后端目标

一期后端的核心目标不是完成真实支付交易闭环，而是先建立一套后续可扩展的支付后台服务骨架。

本期目标包括：

1. 建立 Spring Boot 独立服务
2. 提供工作台、订单、支付单、退款单、服务者结算单的查询接口
3. 固化支付平台的主对象和接口路径
4. 为二期账户账务和清结算能力提供扩展基础

## 3. 模块边界

一期后端只覆盖查询型接口，不包含真实支付受理和真实渠道调用。

当前模块边界如下：

- `DashboardController`：工作台汇总查询
- `OrderController`：订单中心查询
- `PaymentController`：支付单查询
- `RefundController`：退款单查询
- `SettlementController`：服务者结算查询

本期不做：

- 支付下单接口
- 退款发起接口
- 结算发起接口
- 对账处理接口
- 保证金处理接口

## 5. 核心对象

一期需要先固定以下主对象：

1. 订单 `Order`
2. 账单 `Bill`
3. 支付单 `PaymentOrder`
4. 退款单 `RefundOrder`
5. 服务者结算单 `WorkerSettlementOrder`

这些对象虽然当前仍以 mock 数据方式存在，但命名和字段必须提前收敛，避免二期后重构代价过高。

## 6. 接口设计原则

### 6.1 路径规范

采用统一前缀：

`/api`

当前路径：

- `GET /api/dashboard/summary`
- `GET /api/orders`
- `GET /api/payments`
- `GET /api/refunds`
- `GET /api/settlements/workers`

### 6.2 响应规范

当前代码直接返回数组或对象。

后续建议收敛为统一返回体：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "requestId": "REQ202607190001"
}
```

### 6.3 错误处理规范

后续应统一：

- 参数错误
- 资源不存在
- 系统内部异常
- 下游依赖异常

建议在 `common` 包中统一处理：

- `ApiResponse`
- `ErrorCode`
- `BusinessException`
- `GlobalExceptionHandler`

## 7. 数据访问策略

### 7.1 当前阶段

当前阶段采用：

- Service 内部内存 mock 数据

优点：

- 上手快
- 便于快速搭骨架

问题：

- 不利于后续扩展
- 不利于统一数据建模

### 7.2 下一步建议

建议尽快把 mock 数据下沉到 `repository` 层：

- `DashboardMockRepository`
- `OrderMockRepository`
- `PaymentMockRepository`
- `RefundMockRepository`
- `SettlementMockRepository`

这样 service 层职责才会更清晰。

## 8. 业务流说明

### 8.1 订单中心查询流

1. 前端进入订单中心
2. 调用 `GET /api/orders`
3. Controller 接收请求
4. Service 读取订单 mock 数据
5. 返回订单列表

### 8.2 支付单查询流

1. 前端进入支付单管理页
2. 调用 `GET /api/payments`
3. Controller 转发到 Service
4. Service 读取支付单 mock 数据
5. 返回支付单列表

### 8.3 工作台查询流

1. 前端进入工作台
2. 调用 `GET /api/dashboard/summary`
3. Service 聚合指标卡片数据
4. 返回今日指标和风险信息

## 9. 状态设计建议

一期虽然是查询接口，但状态字段必须规范化。

### 9.1 订单状态

建议枚举：

- `WAIT_PAY`
- `WAIT_FULFILL`
- `IN_SERVICE`
- `COMPLETED`
- `CANCELLED`

### 9.2 履约状态

建议枚举：

- `BOOKED`
- `CONFIRMED`
- `SERVING`
- `ACCEPTED`
- `CLOSED`

### 9.3 支付状态

建议枚举：

- `INIT`
- `PAYING`
- `WAIT_CALLBACK`
- `SUCCESS`
- `FAIL`
- `CLOSED`

### 9.4 退款状态

建议枚举：

- `INIT`
- `PROCESSING`
- `SUCCESS`
- `FAIL`

### 9.5 结算状态

建议枚举：

- `INIT`
- `WAIT_AUDIT`
- `WAIT_PAYOUT`
- `PAYOUTING`
- `DONE`

## 10. 可扩展性要求

一期设计时就必须考虑未来扩展：

### 10.1 幂等

后续真实支付、退款、结算接口必须具备幂等能力。

### 10.2 审计

后续所有关键资金操作必须记录：

- 操作人
- 操作时间
- 操作前后值
- 来源系统

### 10.3 重试与补偿

后续真实渠道交互必然存在：

- 查询失败
- 回调丢失
- 状态不一致

因此必须预留重试和补偿机制。

## 11. 与二期、三期的关系

### 11.1 与二期账户账务、清结算的关系

一期的订单、支付单、服务者结算入口，是二期清结算模型的前置数据基础。

### 11.2 与三期退款、对账的关系

一期的支付单和退款单字段收敛，会直接影响三期的：

- 对账口径
- 差异定位
- 退款逆向处理

## 12. 测试建议

后端测试至少要覆盖：

1. 各查询接口返回结构正确
2. 空列表情况下接口仍可正常返回
3. 状态字段值合法
4. 金额字段格式稳定
5. 异常情况下有统一错误返回

## 13. 当前阶段结论

从资深架构视角看，一期后端最重要的不是功能多，而是：

- 路径清晰
- 模型稳定
- 分层明确
- 便于扩展

只要这四点做扎实，后续支付核心、账务、清结算、对账都能沿着同一套框架往下发展。
