# 一期-支付核心系统接口与数据设计

## 1. 文档定位

本文档是 `payment-core` 在 `2026-07-19` 的冻结版 V1 接口与数据设计文档，用于支撑：

- 产品确认页面与接口边界
- 前端按字段完成页面联调
- 后端按对象完成接口实现
- 测试按请求、响应、状态流转执行验证

本文档口径以当前仓库代码为准，强调“已实现什么、如何调用、哪些能力属于后续增强”，不再沿用早期阶段稿口径。

## 2. 系统一期边界

`payment-core` 一期负责家政支付主链路闭环，不负责账务记账、清分清算、出款结算和退款深度编排。

当前冻结版 V1 已纳入范围：

1. 支付工作台查询
2. 订单中心查询与发起支付入口
3. 支付单列表与支付单详情
4. 创建账单与预付单
5. 组织收银台参数
6. 提交支付
7. 模拟渠道回调收口
8. 主动查单
9. 关闭支付单
10. 路由日志、回调日志、事件日志查看
11. 支付请求按请求编号、支付单号、请求状态筛选与分页
12. 支付处理日志按支付单号、阶段、级别筛选与分页
13. 账单、支付流水、收银台会话按交易条件筛选与分页

当前明确不在一期交付范围：

1. 自动记账分录
2. 自动清分与清算出款
3. 退款全流程闭环
4. 对账差异处理闭环
5. 保证金扣罚闭环
6. 多渠道智能路由与失败切换
7. 渠道真实签名验签与真实网关联调

## 3. 页面与接口映射

### 3.1 运营后台 `admin-web`

| 页面 | 路由 | 核心接口 | 说明 |
| --- | --- | --- | --- |
| 工作台 | `/dashboard` | `GET /api/dashboard/summary` | 展示核心经营指标、风险提示 |
| 订单中心 | `/orders` | `GET /api/orders` | 查看订单、识别待支付订单、发起支付 |
| 账单中心 | `/bills` | `GET /api/bills` | 按账单号、订单号、账单状态分页查询交易账单 |
| 支付单管理 | `/payments` | `GET /api/payments` | 查看支付单列表，执行查单、回调、关闭 |
| 支付流水查询 | `/payment-flows` | `GET /api/payment-flows` | 按支付单号、订单号、流水类型分页查询支付过程流水 |
| 收银台会话管理 | `/cashier-sessions` | `GET /api/cashier-sessions` | 按会话号、订单号、终端、会话状态分页查询 |
| 支付请求管理 | `/payment-requests` | `GET /api/payment-requests` | 按请求编号、支付单号、请求状态分页查询，展示终端、客户端 IP 和幂等键 |
| 支付处理日志 | `/payment-logs` | `GET /api/payment-logs` | 按支付单号、处理阶段、日志级别分页查询 |
| 支付单详情 | `/payments/:paymentOrderId` | `GET /api/payments/{paymentOrderId}` | 查看支付主信息、路由轨迹、回调轨迹、事件轨迹 |
| 退款单管理 | `/refunds` | `GET /api/refunds` | 一期保留查询入口，支持退款单号、原支付单号、状态、方式分页排查 |
| 服务者结算单 | `/worker-settlements` | `GET /api/settlements/workers` | 一期保留查询入口，支持结算单号、服务者、结算状态、出款状态分页预览 |

### 3.2 用户端 `app-web`

| 页面 | 路由 | 核心接口 | 说明 |
| --- | --- | --- | --- |
| 收银台 | `/cashier/:prepayOrderNo` | `GET /api/payments/cashier/{prepayOrderNo}` | 展示订单摘要、金额、支付方式，并支持刷新收银台状态 |
| 支付结果页 | `/payment-result/:paymentOrderId` | `GET /api/payments/{paymentOrderId}` | 展示支付成功、处理中、已关闭等结果，并支持查单、模拟回调与轨迹查看 |

### 3.3 支付交易主链路接口

| 业务动作 | 接口 | 方法 | 说明 |
| --- | --- | --- | --- |
| 创建预付单 | `/api/payments/prepay` | `POST` | 基于订单创建账单、支付单、预付单 |
| 获取收银台 | `/api/payments/cashier/{prepayOrderNo}` | `GET` | 查询收银台页面展示数据 |
| 提交支付 | `/api/payments/submit` | `POST` | 选择支付方式后发起支付，记录终端、客户端 IP 和幂等键 |
| 模拟回调 | `/api/payments/callback/{channel}` | `POST` | 收口支付结果并记录回调日志，支持验签、时间窗和 nonce 防重放 |
| 主动查单 | `/api/payments/query` | `POST` | 查询支付单当前状态 |
| 关闭支付单 | `/api/payments/close` | `POST` | 关闭超时或人工终止支付单 |
| 支付单详情 | `/api/payments/{paymentOrderId}` | `GET` | 查询支付详情与全量轨迹 |

## 4. 统一响应模型

所有接口返回统一结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "requestId": "REQ202607190001"
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | string | 业务码，`0` 表示成功 |
| `message` | string | 业务提示 |
| `data` | object/array | 业务数据 |
| `requestId` | string | 请求追踪号 |

## 6. 状态与日志口径

### 6.1 支付主状态

| 对象 | 当前使用状态 |
| --- | --- |
| `t_payment_order.status` | `PREPAY_CREATED` / `SUCCESS` / `WAIT_CALLBACK` / `CLOSED` |
| `t_prepay_order.cashier_status` | `待支付` / `支付中` |
| `t_bill.bill_status` | `待支付` / `已结清` |

### 6.2 轨迹日志

| 日志类型 | 表 | 当前作用 |
| --- | --- | --- |
| 支付尝试日志 | `t_payment_attempt` | 记录支付方式、渠道、终端、客户端 IP、幂等键、请求报文、响应报文 |
| 支付路由日志 | `t_payment_route_record` | 记录路由规则与路由结果 |
| 支付回调日志 | `t_payment_notify_log` | 记录回调类型、回调内容、收口结果 |
| 支付事件日志 | `t_payment_event` | 记录关键业务事件，供后续账务/清结算消费 |

## 7. 开发与联调要求

### 7.1 前端对接要求

1. 页面展示字段必须与接口字段保持一一映射。
2. 状态展示统一使用 `status + statusType` 组合。
3. 详情页必须展示支付单基本信息、回调轨迹、路由轨迹、事件轨迹。
4. 订单中心发起支付后，需能跳转至 `app-web` 收银台。

### 7.2 后端实现要求

1. 分层统一为 `controller / service / serviceimpl / mapper / mapper.xml / entity`。
2. 业务逻辑集中在 `serviceimpl`，`mapper` 仅承载数据读写。
3. `schema.sql` 中所有表和字段必须带 `COMMENT`。
4. `entity` 默认使用 Lombok `@Data`，并保留中文语义注释。
5. 每个系统独立数据库，一期支付核心数据库名为 `housekeeping_payment_core`。

## 8. 当前版本结论

截至 `2026-07-19`，`payment-core` 的接口与数据模型已经具备一期支付交易闭环 V1 的完整主链路字段定义，并完成主要运营查询页的筛选与分页基建，足以支撑前后端继续按冻结版口径开发和联调。

当前版本仍需在后续迭代增强：

1. 继续将更多查询筛选参数真正下推到 SQL，目前已完成支付记录、支付请求、支付处理日志、账单、支付流水、收银台会话、退款单、服务者结算单的基础条件下推
2. 渠道回调验签、防重、冲突收敛
3. 超时关闭巡检任务
4. 更细粒度状态机与异常码体系
5. 对外事件发布和跨系统消费协议
