## 3. 工程规范

### 3.1 后端规范

后端统一采用：

`controller -> service -> serviceimpl -> mapper -> mapper.xml -> entity`

规则：

1. 禁止把业务逻辑放进 `mapper`。
2. 业务逻辑统一放在 `serviceimpl`。
3. `entity` 默认使用 Lombok `@Data`。
4. `entity` 保留数据库语义注释。
5. `schema.sql` 所有表、字段必须带注释。
6. 每个系统独立数据库。

### 3.2 前端规范

1. `admin-web` 承载后台运营能力。
2. `app-web` 承载用户支付链路。
3. 页面结构、字段、状态标签必须与文档保持一致。
4. 所有关键操作必须有成功/失败反馈。

### 3.3 文档规范

1. 文档必须能直接指导开发和 AI 生成代码。
2. 文档要写清楚已实现、未实现和后续增强边界。
3. 所有系统后续都按同一套模板建设。

### 3.4 Maven 执行规范

后端构建与测试统一使用用户指定 Maven 目录，避免不同终端 Maven 版本和本地仓库路径不一致：

```bash
JAVA_HOME=/Library/Java/JavaVirtualMachines/jdk1.8.0_202.jdk/Contents/Home \
/Users/abc123/apache-maven-3.9.16/bin/mvn \
-Dmaven.repo.local=/Users/abc123/apache-maven-3.9.16/repository test
```

执行要求：

1. 不使用临时 `.m2-local` 作为常规验证仓库。
2. 不依赖 shell 默认 `mvn`，必须显式使用 `/Users/abc123/apache-maven-3.9.16/bin/mvn`。
3. 必须显式指定 JDK 目录，避免系统默认 Java 指向 JRE。
4. 测试结果需要同步记录到自测与联调记录。

1. `payment-core` 主链路已跑通
2. 前后端已完成一轮联调
3. 核心文档已进入冻结版
4. 数据库、接口、页面结构已稳定
5. 能为账务、清结算系统提供清晰输入输出边界

## 8. 结论

`payment-core` 是整个平台的资金入口域，后续所有账务、清分、结算、退款、对账能力都要以这里的对象和事件为起点。

因此本次落地要求不是“先搭起来”，而是：

1. 结构要正式
2. 数据要清晰
3. 主链路要闭环
4. 文档要可开发
5. 测试要能验收
