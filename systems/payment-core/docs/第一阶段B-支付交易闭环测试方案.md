# 第一阶段B-支付交易闭环测试方案

## 1. 文档定位

本文档作为 `payment-core` 一期支付交易闭环 V1 的正式测试方案，约束测试范围、测试重点和验收标准。

## 2. 测试目标

确保从订单发起支付到支付结果收口的全过程满足：

1. 主流程可执行
2. 页面与接口字段一致
3. 日志轨迹可追溯
4. 状态变更可验证
5. 旧查询页面不回退

## 3. 测试范围

### 3.1 接口范围

1. 创建预付单
2. 获取收银台
3. 提交支付
4. 模拟回调
5. 主动查单
6. 关闭支付单
7. 支付详情查询

### 3.2 页面范围

1. 工作台
2. 订单中心
3. 支付单管理
4. 支付单详情
5. 收银台
6. 支付结果页

### 3.3 数据范围

1. 支付单
2. 账单
3. 预付单
4. 支付尝试
5. 路由日志
6. 回调日志
7. 事件日志

### 3.4 自动化回归范围

1. 已成功或已关闭支付单的重复/迟到回调不得再次更新状态。
2. 缺少交易状态或渠道交易号的回调必须被拒绝。
3. 超时关单任务必须同时更新支付单、支付尝试、预付单并写入关闭事件。
4. 开启签名校验后，缺少签名字段或签名错误的渠道回调必须被拒绝。

## 4. 核心测试用例

| 用例编号 | 用例名称 | 操作步骤 | 预期结果 | 优先级 |
| --- | --- | --- | --- | --- |
| PAY-TC-001 | 创建预付单成功 | 调用 `POST /api/payments/prepay` | 生成账单、支付单、预付单 | P0 |
| PAY-TC-002 | 获取收银台成功 | 调用 `GET /api/payments/cashier/{prepayOrderNo}` | 返回金额、订单摘要、支付方式 | P0 |
| PAY-TC-003 | 提交支付成功 | 调用 `POST /api/payments/submit` | 预付单进入支付中，写尝试/路由/事件日志 | P0 |
| PAY-TC-004 | 回调成功收口 | 调用 `POST /api/payments/callback/{channel}` | 支付单状态变成功，写回调日志和成功事件 | P0 |
| PAY-TC-005 | 主动查单成功 | 调用 `POST /api/payments/query` | 返回最新支付详情 | P1 |
| PAY-TC-006 | 关闭支付单成功 | 调用 `POST /api/payments/close` | 支付单状态变关闭，写关闭事件 | P1 |
| PAY-TC-007 | 支付详情展示正确 | 调用 `GET /api/payments/{paymentOrderId}` | 返回基本信息、路由、回调、事件轨迹 | P0 |
| PAY-TC-008 | 后台页面回归 | 访问 `admin-web` 页面 | 查询与详情页无阻断错误 | P1 |
| PAY-TC-009 | 收银台回归 | 从订单中心跳转收银台并提交支付 | 页面可完整串联 | P0 |
| PAY-TC-010 | 结果页回归 | 回调后打开结果页 | 成功/处理中/关闭态展示正确 | P1 |
| PAY-TC-011 | 重复回调保护 | 对成功支付单连续回调两次 | 第二次不新增状态更新、日志和事件 | P0 |
| PAY-TC-012 | 超时自动关单 | 将预付单失效时间置为过去并执行任务 | 支付单、支付尝试、预付单和事件同步收口 | P0 |

## 5. 验收标准

1. P0 用例全部通过。
2. 前端构建通过。
3. 页面与接口字段无明显错位。
4. 支付详情页可看到完整轨迹。
5. 支付主链路状态可收口。

## 6. 当前测试边界

当前版本不在本轮必测范围内的能力：

1. 真实第三方网关回调验签
2. 真实查单接口
3. 退款与结算闭环
