# 一期-支付核心系统前端页面与交互说明

## 1. 文档定位

本文档用于指导 `payment-core` 冻结版 V1 的前端页面开发与联调，适用对象：

- 产品经理
- 前端开发
- 测试工程师
- 后端开发

本文档口径以当前已落地的 `admin-web` 与 `app-web` 为准，重点说明页面结构、字段映射、交互动作和当前版本边界。

## 2. 前端交付范围

### 2.1 运营后台 `admin-web`

1. 工作台
2. 订单中心
3. 账单中心
4. 支付流水查询
5. 支付单管理
6. 支付单详情
7. 退款单管理查询页
8. 服务者结算单查询页
9. 统一支付记录
10. 微信支付宝支付记录
11. 银行卡支付记录

### 2.2 用户端 `app-web`

1. 收银台
2. 支付结果页

## 3. 页面布局规范

### 3.1 运营后台

后台统一采用：

1. 左侧导航
2. 顶部页面标题
3. 查询筛选区
4. 表格列表区
5. 操作按钮区
6. 状态标签区

### 3.2 用户端

收银台统一采用：

1. 订单摘要区
2. 金额区
3. 支付方式选择区
4. 提交支付按钮区
5. 结果反馈区

### 3.3 状态色规范

| 语义 | 样式值 | 场景 |
| --- | --- | --- |
| 成功 | `success` | 已支付、已处理、成功 |
| 警告 | `warn` | 待支付、待回调、待处理 |
| 风险 | `danger` | 已关闭、失败、异常 |
| 信息 | `info` | 支付中、处理中、提交中 |

## 4. 页面详细设计

### 4.1 工作台

路由：`/dashboard`

页面目标：

1. 让运营、支付负责人进入系统后快速看到支付健康度
2. 识别待处理风险与当天核心指标

页面区块：

1. 页面标题
2. 风险横幅
3. 四张指标卡
4. 今日重点事项
5. 健康度提示

接口映射：

| 展示项 | 接口字段 |
| --- | --- |
| 卡片标题 | `title` |
| 卡片值 | `value` |
| 角标样式 | `badgeType` |
| 角标文案 | `badgeText` |

交互要求：

1. 首屏自动请求 `/api/dashboard/summary`
2. 卡片区默认展示 4 张
3. 风险横幅固定展示在指标区上方

### 4.2 订单中心

路由：`/orders`

页面目标：

1. 统一查看家政订单支付状态与履约状态
2. 提供发起支付入口
3. 为支付排障提供订单侧入口

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 订单号 | 输入框 | 精确查询 |
| 用户 ID | 输入框 | 当前为前端预留字段 |
| 服务品类 | 下拉框 | 当前为前端预留字段 |
| 订单状态 | 下拉框 | 当前为前端预留字段 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 订单号 | `orderNo` |
| 用户 | `customerName` |
| 服务品类 | `serviceType` |
| 服务者 | `workerName` |
| 订单金额 | `orderAmount` |
| 已付金额 | `paidAmount` |
| 订单状态 | `orderStatus` |
| 履约状态 | `fulfillmentStatus` |
| 创建时间 | `createdAt` |

操作区：

1. 发起支付
2. 查看支付单
3. 查看账单

交互要求：

1. 点击发起支付后调用 `POST /api/payments/prepay`
2. 成功后跳转 `app-web` 收银台
3. 订单状态标签使用接口返回的样式字段

### 4.3 账单中心

路由：`/bills`

页面目标：

1. 按交易账单视角查看订单应收与已收进度
2. 为订单中心和支付单管理提供中间桥梁
3. 明确区分交易账单与账务账簿口径

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 账单号 | 输入框 | 当前以前端过滤为主 |
| 订单号 | 输入框 | 当前以前端过滤为主 |
| 账单状态 | 下拉框 | 当前以前端过滤为主 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 账单号 | `billNo` |
| 订单号 | `orderNo` |
| 客户名称 | `customerName` |
| 账单应收 | `billAmount` |
| 已支付 | `paidAmount` |
| 待支付 | `unpaidAmount` |
| 账单状态 | `billStatus` |
| 到期时间 | `dueAt` |
| 创建时间 | `createdAt` |

交互要求：

1. 首屏自动请求 `GET /api/bills`
2. 当前仅承接交易账单查询，不承接账务会计分录视图
3. 状态标签复用后台统一色值规范

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 支付单号 | 输入框 | 当前以前端过滤为主 |
| 订单号 | 输入框 | 当前以前端过滤为主 |
| 流水类型 | 下拉框 | 当前以前端过滤为主 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 流水编号 | `flowNo` |
| 支付单号 | `paymentOrderId` |
| 订单号 | `orderNo` |
| 预付单号 | `prepayOrderNo` |
| 流水类型 | `flowType` |
| 渠道编码 | `channelCode` |
| 业务状态 | `businessStatus` |

| 创建时间 | `createdAt` |

交互要求：

1. 首屏自动请求 `GET /api/payment-flows`
2. 当前统一聚合支付尝试、渠道回调、路由记录、业务事件四类流水

### 4.5 收银台会话管理

2. 识别终端来源、支付关联和会话失效状态

| 订单号 | 输入框 | 关联家政订单查询 |
| 终端场景 | 下拉框 | H5、PC、APP、小程序 |
| 会话状态 | 下拉框 | 待支付、支付中、支付成功、已完成、已失效 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 会话号 | `sessionNo` |
| 预付单号 | `prepayOrderNo` |
| 支付单号 | `paymentOrderId` |
| 订单号 | `orderNo` |
| 客户 | `customerName` |
| 终端 | `terminal` |
| 金额 | `amount` |
| 会话状态 | `sessionStatus` |
| 创建时间 | `createdAt` |
| 失效时间 | `expiresAt` |

交互要求：

1. 首屏自动请求 `GET /api/cashier-sessions`
2. 会话状态由数据库状态和失效时间综合计算

### 4.13 支付结果页

路由：`/payment-result/:paymentOrderId`

页面目标：

1. 让用户看到支付结果
2. 让测试和运营看到结果是否收口

展示逻辑：

| 支付状态 | 页面反馈 |
| --- | --- |
| `SUCCESS` | 成功态，展示成功提示 |
| `WAIT_CALLBACK` | 处理中，提示稍后刷新 |
| `CLOSED` | 失败/已关闭态，提示支付单已关闭 |
| 其他 | 默认处理中 |

## 5. 联调要求

1. 所有表格页必须保证字段顺序与文档一致。
2. 所有状态标签必须通过 `statusType` 映射样式。
3. 发起支付、提交支付、模拟回调、关闭支付的交互结果必须有成功/失败反馈。
4. `admin-web` 与 `app-web` 必须能完成一次闭环串联。

## 6. 当前版本已知边界

1. 查询筛选当前以前端过滤为主，后续应下推到后端接口。
2. 退款页与服务者结算页当前只作为查询入口，不是完整业务闭环页面。
3. 收银台当前使用模拟渠道，不接真实第三方网关。
4. 页面视觉已具备开发联调价值，但仍可在后续按设计稿继续打磨。

## 7. 结论

从资深前端与产品视角看，当前 `payment-core` 前端已经不是演示页面，而是一套可以支撑产品、前端、后端、测试共同推进的支付核心域 V1 页面交付包。

其价值在于：

1. 页面边界明确
2. 字段映射明确
3. 操作动作明确
4. 联调路径明确
5. 后续扩展位明确
