# 一期-支付核心系统前端页面与交互说明

## 1. 文档定位

本文档是 `一期：支付核心系统 MVP` 的前端设计说明，面向前端开发、产品经理、测试工程师。

本文目标：

- 说明一期前端页面范围
- 说明每个页面该展示什么
- 说明查询区、列表区、详情区的交互方式
- 说明页面字段与接口字段如何映射

## 2. 一期前端范围

一期前端不是完整支付平台，而是 `支付核心后台运营台`。

本期页面清单：

1. 工作台
2. 订单中心
3. 支付单管理
4. 退款单管理入口
5. 服务者结算单入口

说明：

- `退款单管理` 当前是三期退款中心的预留运营视图
- `服务者结算单` 当前是二期清结算系统的预留运营视图

## 3. 整体布局规范

### 3.1 页面结构

后台统一采用：

- 左侧导航
- 右侧内容区
- 页面顶部标题区
- 查询筛选区
- 列表表格区
- 操作按钮区

### 3.2 视觉层级

统一遵循：

- 左侧导航承载模块切换
- 顶部标题承载页面定位和当前重点说明
- 表格为主信息区
- 标签颜色用于表达状态，不承担复杂解释

### 3.3 状态色规范

| 状态类型 | 样式值 | 说明 |
| --- | --- | --- |
| 成功 | `success` | 已完成、支付成功、出款成功 |
| 警告 | `warn` | 待处理、处理中、待审核 |
| 风险 | `danger` | 失败、阻断、异常 |
| 信息 | `info` | 已预约、待出款、进行中 |

## 4. 页面设计

### 4.1 工作台

路由：

`/dashboard`

页面目标：

- 让运营、财务、支付负责人进入系统后第一时间看到支付健康度
- 提示当天重点事项和差异风险

页面结构：

1. 顶部标题区
2. 风险提示横幅
3. 核心指标卡片区
4. 今日重点事项表格
5. 今日健康度表格

指标卡片区字段：

| 展示项 | 说明 |
| --- | --- |
| 今日支付成功金额 | 当日支付收入指标 |
| 今日退款金额 | 当日退款支出指标 |
| 待结算服务者金额 | 待进入结算环节的资金规模 |
| 未关闭差异金额 | 当前未闭环资金差异 |

交互要求：

- 首屏加载时自动请求工作台汇总接口
- 指标卡片默认展示 4 张
- 风险横幅固定展示在卡片区上方
- 今日重点事项按优先级顺序展示

### 4.2 订单中心

路由：

`/orders`

页面目标：

- 统一查看家政订单的支付状态与履约状态
- 为支付问题定位提供订单侧入口

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 订单号 | 输入框 | 支持精确查询 |
| 用户 ID | 输入框 | 支持精确查询 |
| 服务品类 | 下拉框 | 保洁、月嫂、企业保洁等 |
| 订单状态 | 下拉框 | 待支付、待履约、已完成等 |

列表区字段：

| 字段 | 来源接口字段 |
| --- | --- |
| 订单号 | `orderNo` |
| 用户 | `customerName` |
| 服务品类 | `serviceType` |
| 服务者 | `workerName` |
| 订单金额 | `orderAmount` |
| 已付金额 | `paidAmount` |
| 订单状态 | `orderStatus` |
| 履约状态 | `fulfillmentStatus` |
| 创建时间 | `createdAt` |

操作区：

- 详情
- 账单
- 补差价

交互要求：

- 点击查询后带参数刷新列表
- 点击重置后恢复初始条件并重新查询
- 点击订单号后，后续版本可扩展为跳转详情

### 4.3 支付单管理

路由：

`/payments`

页面目标：

- 统一查看支付单状态
- 支撑支付成功、支付处理中、回调未收敛等问题排查

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 支付单号 | 输入框 | 精确查询 |
| 订单号 | 输入框 | 精确查询 |
| 支付方式 | 下拉框 | 微信、支付宝、银行卡 |
| 支付状态 | 下拉框 | SUCCESS、WAIT_CALLBACK、FAIL |

列表区字段：

| 字段 | 来源接口字段 |
| --- | --- |
| 支付单号 | `paymentOrderId` |
| 订单号 | `orderNo` |
| 用户 | `customerName` |
| 实付金额 | `amount` |
| 支付方式 | `paymentMethod` |
| 支付渠道 | `channel` |
| 渠道交易号 | `channelTransactionNo` |
| 支付状态 | `status` |
| 创建时间 | `createdAt` |

操作区：

- 详情
- 主动查询
- 回调

交互要求：

- 支付状态以标签色区分
- 后续版本在“主动查询”点击后可触发后端重新查单
- “回调”后续可扩展为查看渠道回调明细

## 5. 通用交互规范

### 5.1 加载态

- 页面初次进入时显示列表加载态
- 接口返回前禁止重复点击查询

### 5.2 空态

- 无数据时显示空态提示
- 空态文案建议为“当前暂无符合条件的数据”

### 5.3 错误态

- 接口失败时显示错误提示
- 页面保留上次成功结果，避免整页闪空

### 5.4 表格规范

- 列表默认支持横向滚动
- 金额字段右对齐
- 状态字段使用标签，不直接裸文本显示

## 6. 前端组件复用建议

一期建议逐步沉淀以下复用组件：

1. 页面标题区
2. 查询工具栏
3. 状态标签
4. 表格操作栏
5. 空态与错误态

## 7. 测试关注点

前端测试至少覆盖：

1. 路由切换是否正常
2. 接口加载失败时页面是否可用
3. 查询条件重置是否生效
4. 状态标签颜色是否正确
5. 长表格是否可以横向滚动查看

## 8. 结论

一期前端的目标不是做复杂交互，而是先把 `支付核心后台运营台` 做成结构稳定、字段清晰、便于扩展的管理系统。
