# 一期-支付核心系统前端页面与交互说明

## 1. 文档定位

本文档用于指导 `payment-core` 冻结版 V1 的前端页面开发与联调，适用对象：

- 产品经理
- 前端开发
- 测试工程师
- 后端开发

本文档口径以当前已落地的 `admin-web`、`app-web`、`h5-web` 与 `pc-web` 为准，重点说明页面结构、字段映射、交互动作和当前版本边界。

## 4. 页面详细设计

### 4.1 工作台

路由：`/dashboard`

页面目标：

1. 让运营、支付负责人进入系统后快速看到支付健康度
2. 识别待处理风险与当天核心指标

页面区块：

1. 页面标题
2. 风险横幅
3. 四张指标卡
4. 今日重点事项
5. 健康度提示

接口映射：

| 展示项 | 接口字段 |
| --- | --- |
| 卡片标题 | `title` |
| 卡片值 | `value` |
| 角标样式 | `badgeType` |
| 角标文案 | `badgeText` |

交互要求：

1. 首屏自动请求 `/api/dashboard/summary`
2. 卡片区默认展示 4 张
3. 风险横幅固定展示在指标区上方

### 4.2 订单中心

路由：`/orders`

页面目标：

1. 统一查看家政订单支付状态与履约状态
2. 提供发起支付入口
3. 为支付排障提供订单侧入口

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 订单号 | 输入框 | 精确查询 |
| 用户 ID | 输入框 | 当前为前端预留字段 |
| 服务品类 | 下拉框 | 已支持后端筛选，按服务品类精确匹配 |
| 订单状态 | 下拉框 | 已支持后端筛选，按订单状态精确匹配 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 订单号 | `orderNo` |
| 账单号 | `billNo` |
| 用户 | `customerName` |
| 服务品类 | `serviceType` |
| 服务者 | `workerName` |
| 订单金额 | `orderAmount` |
| 已付金额 | `paidAmount` |
| 订单状态 | `orderStatus` |
| 履约状态 | `fulfillmentStatus` |
| 最近支付状态 | `latestPaymentStatus` |
| 最近收银台状态 | `latestCashierStatus` |
| 创建时间 | `createdAt` |

操作区：

1. 发起支付
2. 查看支付单
3. 查看账单
4. 查看支付记录
5. 打开当前收银台

交互要求：

1. 点击发起支付后调用 `POST /api/payments/prepay`
2. 成功后展示最新预付单与收银台入口，可直接打开 `app-web` 收银台
3. 订单状态标签使用接口返回的样式字段
4. 若订单已生成最新支付链路，则支持直接跳到支付单详情、支付记录详情和当前收银台
5. 点击“查看账单”时，账单中心应自动回填 `billNo / orderNo` 查询条件

页面目标：

| 字段 | 类型 | 说明 |

| 账单号 | 输入框 | 已支持后端筛选，按账单号模糊匹配 |
| 订单号 | 输入框 | 已支持后端筛选，按订单号模糊匹配 |
| 客户名称 | 输入框 | 已支持后端筛选，按客户名称模糊匹配 |
| 账单状态 | 下拉框 | 已支持后端筛选，按账单状态精确匹配 |
| 排序字段 | 下拉框 | 已支持创建时间、到期时间、账单应收、待支付金额排序 |
| 排序方向 | 下拉框 | 已支持正序、倒序 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 账单号 | `billNo` |
| 订单号 | `orderNo` |
| 客户名称 | `customerName` |

| 待支付 | `unpaidAmount` |
| 账单状态 | `billStatus` |
| 到期时间 | `dueAt` |
| 创建时间 | `createdAt` |

交互要求：

1. 首屏自动请求 `GET /api/bills`，查询条件通过 `billNo`、`orderNo`、`customerName`、`billStatus`、`sortField`、`sortOrder` 下推到服务端，并支持分页查看
2. 当前仅承接交易账单查询，不承接账务会计分录视图
3. 状态标签复用后台统一色值规范

### 4.4 支付流水查询

路由：`/payment-flows`

页面目标：

1. 聚合展示支付主链路中的关键过程流水
2. 支撑运营和测试查看支付尝试、回调、路由、事件收口过程
3. 让支付排障不再依赖单个详情页逐条翻查

查询区字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 支付单号 | 输入框 | 已支持后端筛选，按支付单号模糊匹配 |
| 订单号 | 输入框 | 已支持后端筛选，按订单号模糊匹配 |
| 流水类型 | 下拉框 | 已支持后端筛选，按流水类型精确匹配 |
| 渠道编码 | 输入框 | 已支持后端筛选，按渠道编码模糊匹配 |
| 终端 | 下拉框 | 已支持 H5、PC、APP、小程序维度筛选 |
| 业务状态 | 下拉框 | 已支持后端筛选，按尝试状态、回调状态、路由结果和事件类型统一筛选 |
| 关键字 | 输入框 | 已支持按摘要、请求报文、响应报文检索 |
| 排序字段 | 下拉框 | 已支持创建时间、重试次数、流水类型排序 |
| 排序方向 | 下拉框 | 已支持正序、倒序 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 流水编号 | `flowNo` |
| 支付单号 | `paymentOrderId` |
| 订单号 | `orderNo` |
| 预付单号 | `prepayOrderNo` |
| 流水类型 | `flowType` |
| 渠道编码 | `channelCode` |
| 业务状态 | `businessStatus` |
| 流水摘要 | `summary` |
| 创建时间 | `createdAt` |
| 操作 | 展开详情、查看支付单详情、查看支付请求、查看处理日志 |

交互要求：

1. 首屏自动请求 `GET /api/payment-flows`，查询条件通过 `paymentOrderId`、`orderNo`、`flowType`、`channelCode`、`terminal`、`businessStatus`、`keyword`、`sortField`、`sortOrder` 下推到服务端，并支持分页查看
2. 当前统一聚合支付尝试、渠道回调、路由记录、业务事件四类流水
3. 点击“展开详情”时，需要展示终端、客户端 IP、幂等键、回调类型、路由规则、下游系统、事件主题、发布状态、重试次数以及原始请求/响应报文
4. 展开详情后，支持直接联查支付单详情、支付请求管理和支付处理日志，减少运营和研发跨页排障成本
5. 状态标签与类型标签复用后台统一样式规范

3. 支撑运营、研发、测试联合排查路由误命中、渠道落错和兜底选择问题

查询区字段：

| 字段 | 类型 | 说明 |

| 订单号 | 输入框 | 已支持后端筛选 |
| 路由规则 | 输入框 | 已支持后端筛选，按规则关键字模糊匹配 |
| 渠道编码 | 输入框 | 已支持后端筛选 |
| 支付方式 | 下拉框 | 已支持微信、支付宝、银行转账维度筛选 |
| 终端 | 下拉框 | 已支持 H5、PC、APP、小程序维度筛选 |
| 路由结果 | 下拉框 | 已支持后端筛选 |
| 排序字段 | 下拉框 | 已支持创建时间、渠道编码、路由结果排序 |
| 排序方向 | 下拉框 | 已支持正序、倒序 |

列表区字段：

| 展示字段 | 接口字段 |
| --- | --- |
| 路由记录号 | `routeNo` |
| 支付单号 | `paymentOrderId` |

交互要求：

2. 点击“展开详情”时，需要展示客户名称、支付金额、终端、客户端 IP、幂等键和最近支付请求/响应报文
3. 展开详情后，支持联查支付单详情、路由流水、支付请求和支付渠道与路由配置页
4. 当前页面定位为支付路由独立排障台，后续可继续补权重、熔断、命中解释和渠道健康度信息

4. 差异告警至少覆盖渠道回调未收口、内部事件未收口、退款待收口三类。
5. 点击差异卡片的“查看”时，应直接跳到异常中心、事件出站页或退款页，缩短排查路径。
6. 手动触发跑批后，页面必须刷新最新批次、对账前置事实、准入判断与差异告警。

### 4.19 支付协议管理

路由：`/payment-config`

页面目标：

1. 统一维护支付协议台账，沉淀协议类型、模板信息、签约要素、协议正文和电子签章服务商。
2. 让产品、运营、研发、测试围绕同一份协议配置数据完成签约规则评审和联调。
3. 为后续电子签章联调、协议版本治理和多协议场景扩展保留稳定结构。

协议表单要求：

1. 协议类型必须从字典中选择，不能手填自由文本。
2. 协议正文必须支持多段文本录入，作为模板正文和签约条款的维护入口。
3. 编辑已有协议时，协议正文、模板信息、签约要素和电子签章服务商必须可完整回填。
4. 列表区必须展示协议类型名称和协议正文，避免只有编码没有业务可读性。

## 5. 联调要求

1. 所有表格页必须保证字段顺序与文档一致。
2. 所有状态标签必须通过 `statusType` 映射样式。
3. 发起支付、提交支付、模拟回调、关闭支付的交互结果必须有成功/失败反馈。
4. `admin-web` 与 `app-web` 必须能完成一次闭环串联。
5. `h5-web` 必须与 `app-web` 保持同一套交易逻辑和接口口径。
6. `pc-web` 必须与 `app-web / h5-web` 保持同一套交易逻辑和接口口径，同时保留桌面端独立文案与交互提示。
7. 支付配置中心当前已补齐“支付渠道配置 + 路由规则配置 + 支付协议管理 + 渠道返回码映射 + 支付网关接入管理”五块内容；返回码映射区至少展示渠道编码、渠道返回码、返回信息、标准状态、标准错误码、是否可重试、是否需要人工介入、处理策略和状态，支持按映射粒度启停；网关区至少展示网关编码、名称、接入模式、适用渠道、基础地址、报文协议、签名算法、超时时间、重试策略和状态。

## 6. 当前版本已知边界

1. 账单中心、支付流水、收银台会话已支持基础后端筛选、分页和运营排序；统一支付记录、微信支付宝支付记录、银行卡支付记录已补齐更多运营筛选与排序；退款单、服务者结算单已支持基础后端筛选，支付请求管理与支付处理日志已支持分页；支付监控分析已补齐摘要卡片、异常告警动作、渠道风险提示和 drill-down 跳转；支付交易异常中心已补齐待回调未收口、回调待处理、事件发布失败、命中停用渠道四类异常聚合与跳转；支付任务中心已补齐自动/手动来源、重点告警、日志筛选分页和严重等级展示；支付日终处理已补齐差异告警卡片与前置收口事实展示；`app-web / h5-web / pc-web` 收银台已支持倒计时、渠道说明、刷新、关闭支付、幂等键重置、支付前检查和失败补救建议展示，支付结果页已支持查单、模拟回调、关闭支付、轨迹查看、下一步建议和补救动作；支付配置中心已补齐协议管理、返回码映射、网关接入管理展示与启停，协议区已支持模板编码/名称、签约要素和电子签章服务商维护；支付事件出站页已补齐下游系统、事件主题、发布状态和重发动作；后续继续补更多运营字段筛选、返回码版本治理和真实渠道能力。
2. 退款页与服务者结算页当前只作为查询入口，不是完整业务闭环页面。
3. 收银台当前使用模拟渠道，不接真实第三方网关。
4. 页面视觉已具备开发联调价值，但仍可在后续按设计稿继续打磨。

## 7. 结论

从资深前端与产品视角看，当前 `payment-core` 前端已经不是演示页面，而是一套可以支撑产品、前端、后端、测试共同推进的支付核心域 V1 页面交付包。

其价值在于：

1. 页面边界明确
2. 字段映射明确
3. 操作动作明确
4. 联调路径明确
5. 后续扩展位明确
