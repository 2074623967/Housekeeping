# payment-core 全量交付差距清单

## 1. 结论

截至 `2026-07-19`，`payment-core` **未达到全量交付标准**，当前状态只能视为：

1. 已完成部分后台运营台查询能力
2. 已初步落地部分支付主链路接口骨架
3. 尚未完成真正可冻结交付的支付核心系统

因此，后续开发顺序必须调整为：

1. 先完成 `payment-core` 全量交付
2. 完成自测、联调、测试用例闭环
3. 再进入 `accounting-system / clearing-system / settlement-system`

## 3. 当前未达到全量交付的核心原因

## 3.1 后端未收口项

1. `entity -> dto` 转换仍未全量规范化，部分 `mapper` 仍直接返回 DTO，不符合最终企业级分层标准。
2. 支付主链路虽然已有接口，但还未证明全部异常流、补偿流、幂等流已真正实现闭环。
3. 回调验签、防重、状态冲突收敛、超时关闭巡检等能力仍未完整落地。
4. 账单状态、支付状态、订单状态之间的联动更新还未做全量规则校验。
5. 缺少完整的接口自测记录和业务流验收记录。

## 3.2 前端未收口项

1. `支付单管理` 页面当前仍偏演示态，查询区未真正接入筛选条件。
2. `支付单详情` 仍是嵌入式详情区，不是正式后台详情页形态。
3. `收银台结果页` 只有简化结果展示，未区分支付成功、失败、处理中的完整交互。
4. `退款单管理` 和 `服务者结算单` 当前仍是预留型入口页，不是完整业务页面。
5. 缺少从订单发起支付到收银台完成支付的完整前端串联入口。

## 3.3 测试未收口项

1. 还没有实际执行并沉淀 `P0 / P1` 测试结果。
2. 缺少接口级测试记录。
3. 缺少前后端联调通过记录。
4. 缺少支付主流程、异常流、补偿流的执行证据。

## 3.4 文档未收口项

1. 部分旧文档仍保留“预留入口”“阶段A/阶段B 未完成”的描述。
2. 文档与当前代码实现之间还没有形成一版“已实现 / 未实现 / 待优化”的最终矩阵。
3. 缺少冻结版交付说明，无法直接作为完整交付包使用。

## 4. 必须补齐的全量交付项

### 4.1 后端必须补齐

1. 完成 `entity + mapper + mapper.xml + service + serviceimpl + controller` 的最终一致性收口。
2. 完成支付主链路状态机收口：
   `订单 -> 账单 -> 预付单 -> 支付提交 -> 回调/查单 -> 成功/关闭`
3. 完成支付异常流：
   订单不存在、预付单不存在、重复回调、回调晚到、已关闭后回调、查单与回调冲突。
4. 完成日志流：
   路由日志、回调日志、事件日志、操作日志。
5. 完成后端自测记录。

### 4.2 前端必须补齐

1. 订单侧增加明确的发起支付入口。
2. 收银台增加更完整的支付方式选择、支付中、支付成功、支付失败页面。
3. 后台支付单详情改造成正式详情页或正式详情弹窗。
4. 支付单管理页支持真实筛选、刷新、查单、关闭、模拟回调联动。
5. 页面字段和接口字段逐项对齐。

### 4.3 测试必须补齐

1. 完成 `P0` 用例执行。
2. 完成 `P1` 用例执行。
3. 输出测试结果文档。
4. 输出回归清单。

### 4.4 文档必须补齐

1. 补最终实现矩阵。
2. 补冻结版交付说明。
3. 补联调和测试结论。

## 5. 进入其他系统开发的前提

只有在以下条件全部满足后，才允许开始其他系统：

1. `payment-core` 主流程可跑通
2. 前后端联调通过
3. 核心测试用例通过
4. 文档进入冻结版
5. 支付核心域具备继续向账务、清分、结算输出事件和接口的能力

从 `2026-07-19` 开始，执行策略调整为：

1. 暂不把 `payment-core` 视为已交付
2. 优先补齐 `payment-core`
3. 补齐后再进入其他系统开发
