# payment-core 全量交付差距清单

## 1. 当前结论

截至 `2026-07-19`，`payment-core` 已经完成“一期支付交易闭环 V1”的主要实现与文档收口，可以视为**进入冻结版交付阶段**，但距离“生产级完全体”仍有增强项。

因此当前判断调整为：

1. 已达到 `开发可执行 + 联调可推进 + 测试可验收` 的交付标准
2. 尚未达到 `真实多渠道生产投产` 的最终标准
3. 可以在此基础上继续做测试收敛和能力增强

## 2. 当前已完成能力

### 2.1 后端

1. `controller / service / serviceimpl / mapper / mapper.xml / entity` 分层已收口
2. 独立数据库 `housekeeping_payment_core` 已定义
3. 核心建表脚本已补表注释与字段注释
4. 查询接口已具备
5. 支付主链路接口已具备
6. 路由日志、回调日志、事件日志已具备

### 2.2 前端

1. `admin-web` 工作台、订单中心、支付单管理、支付单详情已具备
2. `app-web` 收银台、支付结果页已具备
3. 订单到收银台再到支付结果页的串联路径已具备

### 2.3 文档

1. 接口与数据设计已收口
2. 后端详细设计已收口
3. 前端页面与交互说明已收口
4. 数据库设计已收口
5. 接口文档已收口
6. 测试方案与自测记录已补齐

## 3. 当前仍存在的交付差距

### 3.1 生产级增强差距

1. 查询条件尚未全部下推到后端 SQL
2. 回调验签、防重、冲突收敛尚未实现成生产级版本
3. 主动查单尚未接入真实渠道查询
4. 超时关闭巡检任务尚未落地
5. 多渠道路由与失败切换尚未落地

### 3.2 业务域边界差距

1. 退款域当前只有查询入口，没有完整退款闭环
2. 结算域当前只有查询入口，没有审批、出款、核销闭环
3. 账务、清分、清算、对账仍需后续系统承接

### 3.3 测试执行差距

1. 前端构建验证已完成，但后端完整 Maven 编译仍受本地依赖拉取限制
2. 后端需要补持续性的接口自动化测试
3. 需要补更完整的异常流回归记录

## 4. 当前版本是否可交付

从资深产品经理、资深架构师、资深测试经理视角看：

- 如果目标是“作为公司内冻结版开发包继续推进”，当前版本可以交付。
- 如果目标是“直接对接真实支付渠道上线生产”，当前版本还不可以直接投产。

## 5. 后续优先级建议

### P0

1. 完成后端运行验证与接口自测补强
2. 补查询参数下推
3. 补关闭、回调、查单的异常流校验

### P1

1. 引入真实渠道适配层
2. 补回调验签、防重、补偿任务
3. 建立支付事件对下游账务/清结算的标准输出协议

### P2

1. 推进退款中心
2. 推进清结算系统
3. 推进对账与保证金系统

## 6. 结论

`payment-core` 当前最重要的事情，不再是“证明它有没有搭起来”，而是基于这版冻结稿继续做：

1. 运行验证
2. 测试补强
3. 生产级增强
4. 下游系统衔接
