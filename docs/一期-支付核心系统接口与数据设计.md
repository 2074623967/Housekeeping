# 一期-支付核心系统接口与数据设计

## 1. 文档定位

本文档是 `家政支付系统` 一期开发文档，范围仅覆盖 `支付核心系统 MVP`。

适用对象：

- 产品经理
- 前端开发
- 后端开发
- 测试工程师

本文目标：

- 明确一期接口边界
- 明确一期页面与接口映射关系
- 明确一期最小数据模型
- 让开发知道当前阶段该做什么、不该做什么

## 2. 一期范围

一期只覆盖以下 5 个能力：

1. 支付运营工作台
2. 订单中心查询
3. 支付单管理查询
4. 退款中心入口查询
5. 服务者结算入口查询

一期不做：

- 支付下单正式受理
- 渠道签约
- 渠道真实回调落库
- 自动记账
- 自动清分
- 自动出款
- 对账差异闭环
- 保证金扣罚闭环

## 3. 页面与接口映射

| 页面 | 路由 | 接口 | 说明 |
| --- | --- | --- | --- |
| 工作台 | `/dashboard` | `GET /api/dashboard/summary` | 查询今日核心经营与风险指标 |

| 支付单管理 | `/payments` | `GET /api/payments` | 查询支付单列表 |
| 退款单管理 | `/refunds` | `GET /api/refunds` | 查询退款单列表，当前为退款中心预留入口 |

## 4. 接口统一约定

虽然当前代码还是直接返回数组或对象，但从一期规范开始，接口统一建议收敛为以下格式：

```json
{
  "code": "0",
  "message": "success",
  "data": {},
  "requestId": "202607190001"
}
```

说明：

- `code`：业务响应码，`0` 表示成功
- `message`：接口说明
- `data`：真实业务数据
- `requestId`：链路追踪号

分页列表建议统一为：

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "list": [],
    "pageNo": 1,
    "pageSize": 20,
    "total": 0
  },
  "requestId": "202607190001"
}
```

## 7. 后端代码分层建议

当前已有：

- `controller`
- `service`

下一步建议补齐：

1. `dto`：入参、出参对象
2. `enums`：状态枚举、支付方式枚举
3. `repository`：mock repository 或数据库访问层
4. `common`：统一返回体、异常、响应码

建议目标结构：

```text
backend/src/main/java/com/abc123/hsp
├── common
├── controller
├── dto
├── enums
├── repository
├── service
└── HomeServicePaymentApplication.java
```

## 8. 前端开发约定

当前前端页面已经能承接一期范围，但还需要遵守以下约束：

1. 一期只做后台管理台，不做用户端收银台
2. 页面字段必须和接口字段保持一致
3. 查询区域、表格区、操作区的结构要可复用
4. 先补空态、错误态、加载态，再扩页面

## 9. 开发顺序建议

建议当前项目继续按下面顺序推进：

1. 统一后端返回体
2. 将现有 mock 数据从 service 内部拆到 repository
3. 前端适配统一响应结构
4. 补列表页空态、异常态、加载态
5. 固化订单中心和支付单管理页
6. 再进入二期账户账务与结算建设

## 10. 结论

从开发执行角度看，当前项目最重要的不是继续横向铺更多系统，而是先把 `一期支付核心系统` 做扎实。

只要这一期把：

- 页面骨架
- 查询接口
- 主对象模型
- 返回体规范
- 代码分层规范

固定下来，后续二期、三期的扩展成本会明显下降。
