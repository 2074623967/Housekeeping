## 3. 平台建设目标

目标不是只做一个“支付页面”或者“后台列表”，而是建设一套可支撑家政业务完整资金链路的平台。

目标能力包括：

1. 收款
2. 退款
3. 充值
4. 提现
5. 转账
6. 账户记账
7. 清分清算
8. 结算出款
9. 对账差错
10. 保证金
11. 钱包与红包
12. 二清与合规监管

## 4. 总体系统拆分

结合支付系统、账户系统、对账系统、保证金系统、钱包系统母稿，最终拆分为以下核心系统：

## 6. 前端总体架构

从资深前端架构视角看，前端不能只做一个后台项目，而要按端拆分。

### 6.1 前端端形态

1. `admin-web`
2. `app-web`
3. `h5-cashier`
4. `open-api-console`

### 6.2 当前 payment-core 的前端策略

当前 `systems/payment-core/frontend` 已拆分为：

```text
frontend
├── admin-web
├── app-web
└── shared
```

说明：

- `admin-web`：后台管理台
- `app-web`：用户端 Web/H5 预留
- `shared`：共享模型、工具和常量

## 7. 后端总体架构

从资深后端架构视角看，系统要按业务域拆，而不是所有能力压进一个大服务。

推荐结构：

```text
systems
├── payment-core
├── accounting-system
├── clearing-system
├── settlement-system
├── refund-center
├── reconciliation-system
├── deposit-system
├── wallet-system
├── risk-control-system
└── ops-config-system
```

每个系统内部再维持：

```text
system-name
├── docs
├── frontend
├── backend
└── database
```

## 8. 数据库总体设计原则

### 8.1 数据库不是单库大表堆砌

要按领域建模，至少要区分：

- 支付域
- 账户域
- 清结算域
- 对账域
- 保证金域
- 钱包域

### 8.2 当前推荐库级拆分

1. `housekeeping_payment`
2. `housekeeping_accounting`
3. `housekeeping_clearing`
4. `housekeeping_reconciliation`
5. `housekeeping_deposit`
6. `housekeeping_wallet`

一期 `payment-core` 先使用：

- `housekeeping_payment`
