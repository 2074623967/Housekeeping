# 四类资料驱动开发规范

## 1. 适用范围

本规范适用于家政服务支付平台后续所有系统、前端、后端、数据库、接口、测试和交付工作。

## 2. 四类资料的职责

| 资料来源 | 主要作用 | 开发时必须产出的内容 |
| --- | --- | --- |
| `/Users/abc123/图稿` | 获取系统结构、页面形态、菜单层级、模块边界、字段密度和交互入口 | 页面矩阵、页面原型还原、字段清单、端类型说明 |
| `/Users/abc123/笔记` | 获取业务背景、业务流程、功能拆解、状态流转和接口思路 | PRD、状态机、业务流程、异常分支、验收标准 |
| `/Users/abc123/workspace/支付相关` | 获取已经沉淀的方案、DDL、OpenAPI、前后端设计和历史决策 | 系统详细设计、数据库设计、接口文档、联调说明 |
| `chentianyuzhou.com` | 补充支付专家视角的方法论、系统拆分、支付控制和资金流程 | 架构评审、边界调整、幂等/重试/补偿/对账/日终等专业补充 |

## 3. 标准工作流

每个新系统或新需求必须按以下顺序执行：

1. 资料读取：先读取四类资料中与当前系统相关的页面、流程和方案。
2. 需求归并：建立“原始资料 -> 业务能力 -> 系统归属 -> 页面/接口/数据”的映射表。
3. 架构评审：从财务产品、后端架构、前端、测试四个角色检查职责边界。
4. 文档冻结：先完成需求文档、前端页面与交互说明、后端详细设计、接口文档、数据库设计和测试方案。
5. 前后端实现：前端按图稿页面形态实现，后端按接口和状态机实现，数据库字段必须有业务注释。
6. 联调验证：验证接口契约、页面交互、数据库写入、核心业务状态流转和异常分支。
7. 专业优化：对比成熟支付产品的通用做法，将有价值的优化登记到需求清单并迭代实现。
8. Git 交付：需求使用 `feature/需求名`，缺陷使用 `hotfix/需求名-缺陷`，完成自测后合并到 `test`，验证通过后合并到 `master`，按版本发布 `release/*`。

## 4. 前端实现约束

1. 图稿中的页面标题、菜单层级、查询字段、表格列、操作列和状态展示必须逐项核对，不能用摘要卡片替代原型表格。
2. PC 后台、PC 用户端、H5、APP、小程序必须明确端边界，不能把多个端混成一个页面。
3. 页面必须覆盖加载中、空数据、接口失败、成功、处理中和失败状态。
4. 页面字段命名必须和接口字段建立映射，复杂表格必须保留横向滚动和字段密度。
5. 页面完成后至少验证默认态、查询态、重置态、详情跳转和响应式/最小宽度行为。

## 5. 后端实现约束

1. 默认采用 `controller / service / serviceimpl / dao / mapper / mapper.xml` 分层，按当前系统实际使用情况落地。
2. Controller 只负责参数绑定、校验和响应封装，业务编排放在 `serviceimpl`。
3. 所有状态变化必须明确前置状态、目标状态、幂等条件、失败处理和补偿方式。
4. 支付、退款、出款、入账、清分和对账必须区分业务事实，禁止跨域直接修改其他系统核心表。
5. SQL、Entity、DTO、接口字段和页面字段保持同一业务含义；建表字段必须有 `COMMENT`。
6. 重要接口、状态机、重试、回调验签、幂等、防重、锁和事务边界必须写注释和设计说明。

## 6. 测试与交付约束

1. 每个需求必须有正常流程、重复请求、参数错误、数据不存在、状态冲突、外部渠道失败和超时场景用例。
2. 核心流程必须做接口串联测试，不能只验证单个接口返回 200。
3. 前端构建、后端编译、数据库初始化和关键 API 冒烟测试通过后才允许进入 `test`。
4. 测试未通过不得合并 `master`，未形成版本记录不得创建 `release`。
5. 每次提交必须写清变更范围、验证命令、已知限制和后续需求。
