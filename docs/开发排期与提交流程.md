生效日期：

- `2026-07-19`

## 2. 执行原则

从 `2026-07-19` 开始，项目按以下顺序推进：

1. 先排期
2. 再做需求与设计
3. 再做代码实现

## 4. 模块开发顺序

每个模块都必须按如下顺序执行：

1. 确认排期
2. 明确模块范围
3. 输出数据库设计
4. 输出前端页面设计
5. 输出后端详细设计
6. 输出接口文档
7. 开始代码开发
8. 做本地验证
9. 提交 Git
10. 推送 GitHub

## 5. Git 提交节奏

后续不采用“大改一堆最后一次性提交”的方式，而采用小步提交。

建议提交粒度：

1. 文档基线提交
2. 目录结构提交
3. 数据库设计提交
4. 后端接口提交
5. 前端页面提交
6. 联调修复提交

提交信息建议：

- `docs: ...`
- `feat: ...`
- `refactor: ...`
- `fix: ...`
- `chore: ...`

## 6. Git 分支与发布流程

从 `Sunday, July 19, 2026` 开始，项目采用以下分支策略：

### 6.1 分支职责

- `test`
  - 日常开发、自测、联调默认在该分支推进
  - 新需求、新系统、新改造先进入 `test`
- `master`
  - 只承接已经完成自测并确认可发布的代码
  - 作为稳定主干
- `release/*`
  - 在需要发版时，从 `master` 按版本号切出发布分支
  - 用于版本冻结、最终检查和打包

### 6.2 分支命名规范

- 正常需求：
  - `feature/需求名`
- 缺陷修复：
  - `hotfix/需求名-缺陷`
- 发布分支：
  - `release/版本号`

示例：

- `feature/payment-core-phase-b`
- `feature/accounting-system-v1`
- `hotfix/payment-core-callback-dup-fix`
- `release/v1.0.0`

### 6.3 标准流程

标准流程必须是：

1. 需求设计与代码开发先进入 `test`
2. 具体开发在 `feature/*` 或 `hotfix/*` 分支进行
3. 完成后先合入 `test`
4. 在 `test` 完成自测
5. 自测通过后再进入 `master`
6. 基于 `master` 在某个时刻切 `release/*`
7. 在 `release/*` 上完成发布包和版本号管理

### 6.4 当前执行要求

后续我在仓库中的默认工作方式是：

1. 先按排期做文档设计
2. 再从 `test` 拉出对应的 `feature/*` 或 `hotfix/*`
3. 在该分支推进代码
4. 完成本地验证和自测
5. 先合入 `test`
6. 自测通过后再推进到 `master`
7. 发布时再配合生成 `release/*` 版本

## 7. 当前约束

当前 `payment-core` 的开发也必须服从这份排期文档。

也就是说，后续需求开发、设计、实现和 Git 提交，都以本文件为基准执行。

## 8. 持续优化规则

每完成一轮设计、开发或测试后，都要额外做一次对标检查：

1. 对照市面同类产品
2. 对照当前母稿文档
3. 判断是否有更优实现
4. 如果有，则先加入对应系统的需求列表
5. 再基于需求列表继续完成设计、开发、测试和提交

要求：

- 不是发现更优实现后直接口头记住
- 而是必须先进入系统需求列表
- 再按列表进入下一轮排期和开发
