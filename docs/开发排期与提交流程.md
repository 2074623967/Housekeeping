# 开发排期与提交流程

## 1. 文档定位

本文档用于固定 `home-service-payment-system` 的开发排期、模块推进顺序和 Git 提交节奏。

生效日期：

- `2026-07-19`

## 2. 执行原则

从 `2026-07-19` 开始，项目按以下顺序推进：

1. 先排期
2. 再做需求与设计
3. 再做代码实现
4. 最后做 Git 提交与 GitHub 推送

后续不允许脱离排期零散开发。

## 3. 总排期

### 3.1 第一阶段：支付核心系统一期 MVP

时间范围：

- `2026-07-19` 至 `2026-07-26`

目标：

- 完成 `systems/payment-core` 的一期可开发版本
- 完成数据库设计、前端设计、后端设计、接口文档、代码实现
- 打通 admin-web、后端接口、MySQL

阶段任务：

1. `2026-07-19`
   - 项目初始化
   - Git 建仓
   - GitHub 绑定
   - 系统目录拆分
2. `2026-07-19` 至 `2026-07-20`
   - 补齐 payment-core 文档
   - 拆分前端端结构
   - 确认数据库和运行环境
3. `2026-07-20` 至 `2026-07-22`
   - 接入 MySQL
   - 完成建表、初始化数据
   - 完成后端统一返回体和查询接口改造
4. `2026-07-22` 至 `2026-07-24`
   - admin-web 适配新接口
   - 补齐加载态、错误态、空态
   - 收敛后台页面结构
5. `2026-07-24` 至 `2026-07-26`
   - 联调
   - 修复问题
   - 输出一期阶段总结

### 3.2 第二阶段：账户账务与清结算

计划时间：

- `2026-07-27` 至 `2026-08-09`

目标：

- 新增 `accounting-system`
- 新增 `clearing-system`
- 建立账户、流水、清分、结算能力

### 3.3 第三阶段：退款中心、对账、保证金

计划时间：

- `2026-08-10` 至 `2026-08-23`

目标：

- 新增 `refund-center`
- 新增 `reconciliation-system`
- 新增 `deposit-system`

### 3.4 第四阶段：钱包、营销、线下汇入、业财协同

计划时间：

- `2026-08-24` 之后

目标：

- 完成完整资金平台扩展能力

## 4. 模块开发顺序

每个模块都必须按如下顺序执行：

1. 确认排期
2. 明确模块范围
3. 输出数据库设计
4. 输出前端页面设计
5. 输出后端详细设计
6. 输出接口文档
7. 开始代码开发
8. 做本地验证
9. 提交 Git
10. 推送 GitHub

## 5. Git 提交节奏

后续不采用“大改一堆最后一次性提交”的方式，而采用小步提交。

建议提交粒度：

1. 文档基线提交
2. 目录结构提交
3. 数据库设计提交
4. 后端接口提交
5. 前端页面提交
6. 联调修复提交

提交信息建议：

- `docs: ...`
- `feat: ...`
- `refactor: ...`
- `fix: ...`
- `chore: ...`

## 6. 当前约束

当前 `payment-core` 的开发也必须服从这份排期文档。

也就是说，后续需求开发、设计、实现和 Git 提交，都以本文件为基准执行。
