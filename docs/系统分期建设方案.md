# 家政支付系统分期建设方案

## 1. 文档定位

本文档用于明确家政支付系统的开发顺序、系统边界和每一期交付目标，避免项目一开始就陷入“页面很多、模块很多、优先级混乱”的问题。

项目原则：

- 先按 `系统域` 拆分，再按 `阶段` 落地。
- 每一期都必须形成 `可运行、可验证、可继续扩展` 的完整闭环。
- 不追求一次性做完整平台，追求每一期都能沉淀真实可复用的底座。
- 每个系统默认独立数据库，避免交易、账务、清分、结算等核心域共库耦合。

## 2. 系统拆分总览

结合家政业务和已有方案母稿，支付平台拆分为以下系统域：

1. 支付核心系统
2. 账户账务系统
3. 清分清算系统
4. 结算出款系统
5. 退款中心
6. 对账系统
7. 保证金系统
8. 钱包与营销资金系统
9. 线下汇入与对公收款系统
10. 支付配置中心
11. 风控与差错处理中心
12. 业财税票协同接口层

## 3. 开发顺序

### 3.1 一期：支付核心域 MVP

目标：

- 打通家政订单的基础支付主链路
- 建立后台运营最小工作台
- 建立订单、账单、支付单的查询和联动关系
- 为后续退款、账务、结算提供稳定入口

建设范围：

- 支付运营工作台
- 订单中心
- 支付单管理
- 支付状态查询
- 支付渠道结果回写
- 支付单与订单联动展示
- 基础支付查询 API
- 前后端工程骨架

本期不做：

- 完整账务分录
- 完整结算链路
- 自动清分
- 财务对账闭环
- 保证金扣罚闭环

当前代码状态：

- `/Users/abc123/workspace/home-service-payment-system/systems/payment-core/frontend`
- `/Users/abc123/workspace/home-service-payment-system/systems/payment-core/backend`

这部分就是当前正在开发的一期骨架。

数据库规划：

- `payment-core` 独立库：`housekeeping_payment_core`

### 3.2 二期：账户账务与清结算域

目标：

- 把支付成功后的资金归属、平台抽佣、服务者应收、商家应收明确下来
- 建立从支付结果到内部账务再到待结算结果的闭环

建设范围：

- 账户体系
- 账户余额、冻结、解冻、可提现
- 账务流水与记账分录
- 清分规则
- 服务者结算单
- 商家结算单
- 打款批次
- 清结算任务

交付结果：

- 家政平台可以开始支撑真实服务者结算试算
- 财务可以开始验证抽佣、补贴、结算金额是否正确

数据库规划：

- `accounting-system` 独立库：`housekeeping_accounting`
- `clearing-system` 独立库：`housekeeping_clearing`
- `settlement-system` 独立库：`housekeeping_settlement`

### 3.3 三期：退款、对账、保证金与运营配置域

目标：

- 让系统具备逆向交易、差错处理和财务核对能力

建设范围：

- 退款中心
- 退转付
- 对账批次与差异处理
- 渠道账单导入
- 平台账与渠道账核对
- 保证金收取、冻结、扣罚、退还、欠款抵扣
- 运营后台配置中心

交付结果：

- 支付平台从“能收款、能结算”升级为“能核对、能纠错、能兜底”

数据库规划：

- `reconciliation-system` 独立库：`housekeeping_reconciliation`
- `deposit-system` 独立库：`housekeeping_deposit`
- `refund-center` 独立库：`housekeeping_refund`

### 3.4 四期：钱包、营销、线下汇入、二清监管与业财协同

目标：

- 支撑更复杂的业务资金场景和跨系统财务协同

建设范围：

- 钱包账户
- 红包、优惠券、补贴资金
- 线下汇入支付单
- 对公收款与认领
- 二清监管户场景
- 发票、税务、会计协同接口
- 资金日报、经营报表

交付结果：

- 形成完整的家政资金平台

数据库规划：

- `wallet-system` 独立库：`housekeeping_wallet`
- `offline-payment-system` 独立库：`housekeeping_offline_payment`
- `payment-config-center` 独立库：`housekeeping_payment_config`
- `risk-center` 独立库：`housekeeping_risk`
