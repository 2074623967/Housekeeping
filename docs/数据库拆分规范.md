# 数据库拆分规范

## 1. 规范目标

本文档用于明确家政支付平台的数据库拆分原则，避免多个系统共用同一个数据库导致：

1. 表边界混乱
2. 领域职责不清
3. 迁移风险增大
4. 权限隔离困难
5. 运维排障复杂

## 2. 基本原则

从 `2026-07-19` 开始，平台内每个系统默认采用：

1. 独立数据库
2. 独立初始化脚本
3. 独立表前缀和对象边界
4. 独立连接配置

不采用：

1. 多系统共库
2. 一个库里混放交易、账务、清分、结算、保证金等核心域表

## 3. 当前数据库规划

| 系统 | 数据库名 | 说明 |
| --- | --- | --- |
| `payment-core` | `housekeeping_payment_core` | 支付核心交易域 |
| `accounting-system` | `housekeeping_accounting` | 账户与账务域 |
| `clearing-system` | `housekeeping_clearing` | 清分清算域 |
| `settlement-system` | `housekeeping_settlement` | 结算出款域 |
| `refund-center` | `housekeeping_refund` | 退款逆向交易域 |
| `reconciliation-system` | `housekeeping_reconciliation` | 对账差异处理域 |
| `deposit-system` | `housekeeping_deposit` | 保证金与欠款抵扣域 |
| `wallet-system` | `housekeeping_wallet` | 钱包与营销资金域 |
| `offline-payment-system` | `housekeeping_offline_payment` | 线下汇入与对公收款域 |
| `payment-config-center` | `housekeeping_payment_config` | 支付配置中心 |
| `risk-center` | `housekeeping_risk` | 风控与差错处理域 |

## 4. 实施要求

### 4.1 配置要求

每个系统必须有自己的：

1. `application.yml` 数据库连接
2. `schema.sql`
3. `data.sql`
4. 数据库设计文档

### 4.2 开发要求

后续所有系统开发默认按以下方式执行：

1. 先确认系统边界
2. 再确定独立数据库名
3. 再设计库表和字段注释
4. 再接入后端配置

### 4.3 交付要求

凡是进入“可交付”状态的系统，都必须明确：

1. 使用哪个数据库
2. 表属于哪个系统
3. 是否允许跨系统读写

## 5. 当前落地结论

`payment-core` 先按独立库 `housekeeping_payment_core` 推进。

后续 `accounting / clearing / settlement / refund / reconciliation / deposit / wallet` 等系统，全部沿用本规范，不再共用一个数据库。
