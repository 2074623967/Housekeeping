# 发布门槛与版本治理规范

## 1. 文档定位

本文档用于约束家政服务支付系统从 `feature/*` 到 `test`、从 `test` 到 `master`、再到 `release/*` 的正式发布门槛，避免“代码做了很多，但没有统一发版标准”的情况。

## 2. 分支推进门槛

### 2.1 `feature/*` 合入 `test`

必须同时满足：

1. 对应系统的需求文档、接口文档、前端交互说明、数据库设计至少完成本轮变更覆盖。
2. 代码实现完成，并与既有技能规范一致：
   - 后端遵循 `java-ponytail`
   - 前端遵循 `frontend-craftsman`
3. 本轮涉及的后端测试通过。
4. 本轮涉及的前端构建通过。
5. 自测记录已回填到对应系统文档。
6. `git diff --check` 通过。

### 2.2 `test` 合入 `master`

必须同时满足：

1. 本轮需求已达到“可给产品、前端、后端、测试直接使用”的冻结版交付标准。
2. 对应系统关键链路已完成至少一轮端到端自测。
3. `test` 分支无阻断级已知问题。
4. 核心文档齐全：
   - 需求/范围说明
   - 前端页面与交互说明
   - 后端详细设计
   - 接口文档
   - 数据库设计
   - 自测与联调记录
   - 当前交付状态说明
5. 同类产品对标后，没有明确遗漏但应进入本轮的 P0/P1 项。
6. 合并说明中写清楚版本范围、影响系统、风险点和回滚边界。

### 2.3 `master` 切 `release/*`

必须同时满足：

1. `master` 已经完成对应版本范围内的稳定收口。
2. 发布文档已经准备完成。
3. 数据库变更、接口变更、前端入口变更已明确。
4. 已形成已知风险清单。
5. 已形成发布后验证清单与回滚说明。

## 3. 发布文档要求

每次创建 `release/*` 时，必须在 `docs/releases/` 下新增对应版本文档，至少包括：

1. 版本号
2. 发布日期
3. 发布范围
4. 本次功能清单
5. 数据库变更
6. 接口变更
7. 前端变更
8. 测试结论
9. 已知风险
10. 回滚说明
11. 发布后验证项

## 4. 当前执行原则

截至 `Tuesday, July 21, 2026`：

1. `feature/payment-core-phase-b` 仍明显领先于 `test/master`，说明当前还处于持续正式化阶段。
2. 在 `payment-core` 还未达到完整冻结版门槛前，不应机械合入 `master`。
3. 后续自动化推进时，应优先继续补齐关键缺口，再择机推进 `test -> master -> release/*`。

## 5. 自动合并与自动发布判定规则

后续允许由 Codex 自动推进 `test -> master -> release/*`，但必须严格遵守以下判定，不以“开发做了很多”为依据，而以“冻结版质量是否达标”为依据。

### 5.1 允许自动合入 `master` 的条件

必须同时满足：

1. 当前 `feature/*` 对应的 P0/P1 范围已经形成阶段性闭环，不再是明显半成品。
2. `test` 分支已经包含本轮代码，且最新验证结果完整：
   - 后端自动化测试通过
   - 前端相关端构建通过
   - 自测与联调记录已更新
3. 对应系统文档已达到可直接给产品、前端、后端、测试、AI 开发共同使用的标准。
4. 当前已知问题中不存在阻断级缺陷、核心链路断裂、关键页面缺失、关键接口未实现这四类问题。
5. 同类产品对标后，本轮应该补进来的 P0/P1 项已经补齐，剩余缺口明确属于后续迭代而非当前冻结版阻断项。

### 5.2 允许自动创建 `release/*` 的条件

必须同时满足：

1. `master` 已完成一轮稳定观察，未发现新增阻断问题。
2. 版本范围、数据库变更、接口变更、前端入口变更、风险点、回滚方案均已写入发布文档。
3. 发布后验证项已经列明，并能覆盖核心支付链路、退款链路、配置链路和异常处理链路。
4. 本次版本可以被视为一个完整可交付里程碑，而不是单个零散功能点。

### 5.3 当前判断

截至 `2026-07-21`：

1. 还不满足自动合入 `master` 条件。
2. 还不满足自动创建 `release/*` 条件。
3. 当前正确策略仍是继续在 `feature/payment-core-phase-b` 上推进 `payment-core` 正式版冻结工作，完成更多 P0/P1 能力后再评估分支晋级。
