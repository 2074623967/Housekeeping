# 模块开发交付标准

## 1. 文档定位

本文档用于固定本项目后续每一个系统模块的开发交付标准。

从本文件生效开始，任何模块开发都不能只做代码，必须同步交付完整设计文档。

## 2. 每个模块必须交付的内容

后续无论开发 `支付核心`、`账户账务`、`清结算`、`退款中心`、`对账系统`、`保证金系统` 还是其他模块，都必须同时包含以下 5 类输出：

1. 数据库设计
2. 前端页面设计
3. 后端设计
4. 接口文档
5. 实际代码开发

## 3. 具体要求

### 3.1 数据库设计

必须包含：

- 表清单
- 表用途说明
- 核心字段字典
- 主键、唯一键、索引设计
- 状态字段说明
- 表与表之间的关系
- 是否需要分库分表或归档

交付物建议：

- `xxx模块后端接口与数据库设计.md`
- 必要时补充 `DDL.sql`

### 3.2 前端页面设计

必须包含：

- 页面清单
- 每个页面的用途
- 查询区、列表区、详情区、弹窗区说明
- 关键交互流程
- 页面字段与接口字段映射
- 状态展示和异常展示说明

交付物建议：

- `xxx模块前端页面与交互说明.md`

### 3.3 后端设计

必须包含：

- 模块职责边界
- 核心对象模型
- 状态机或状态流转
- 任务设计
- 异常处理
- 幂等、重试、补偿
- 审计和日志要求

交付物建议：

- `xxx模块详细设计.md`

### 3.4 接口文档

必须包含：

- 接口清单
- 请求路径
- 请求方法
- 请求参数
- 返回参数
- 错误码
- 样例报文

交付物建议：

- `xxx模块接口文档.md`
- 必要时补充 `OpenAPI.yaml`

### 3.5 实际代码开发

必须包含：

- 前端页面代码
- 后端接口代码
- 基础 mock 或真实数据接入
- 关键枚举、DTO、repository、service、controller 分层

## 4. 标准开发顺序

每个模块都按以下顺序推进：

1. 先写模块目标和业务范围
2. 再写数据库设计
3. 再写前端页面设计
4. 再写后端设计
5. 再写接口文档
6. 最后进入代码开发与联调

## 5. 当前项目执行规则

本项目后续所有系统开发都遵循以下规则：

- 不能只补一个页面
- 不能只补几条接口
- 不能只写一份抽象方案
- 必须形成“文档可开发、接口可对接、代码可运行”的完整交付包

## 6. 当前一期的落实方式

对于当前 `一期：支付核心系统 MVP`，后续也将按这个标准补齐：

- 支付核心数据库设计
- 支付核心前端页面设计
- 支付核心后端设计
- 支付核心接口文档
- 支付核心代码实现

## 7. 结论

后续本项目的开发单位，不再是“单个页面”或“单个接口”，而是“完整模块交付包”。

这也是为了确保文档可以直接给产品、前端、后端、测试，甚至 AI 开发工具直接使用。
